CFA 1.1.0 is out. It is intentionally a small release. No new features —
just the work the project needed before earning the right to add features.

<!-- truncate -->

The five primitives that make CFA distinctive are unchanged:

1. Typed, content-hashed `StateSignature`.
2. `REPLAN` as a first-class policy outcome, not just yes/no.
3. SHA-256 audit chain, verifiable offline.
4. Operational catalog (PII, partition, classification, merge key as
   policy primitives — not just search metadata).
5. Deterministic-by-default decisions; LLM as an opt-in extra.

Each of those now has an ADR explaining the call. See
[`docs/adr/`](https://github.com/marquesantero/cfa/tree/main/docs/adr).

## What changed

### Package consolidation (20 → 16 subpackages)

The 1.0.0 layout shipped with two pairs of overlapping packages and one
pure re-export facade. They're gone in 1.1.0:

- `cfa.normalizer` + `cfa.resolution` → **`cfa.resolve`**.
- `cfa.governance` → **absorbed into `cfa.policy`**.
- `cfa.validation` → **`cfa.validate`**.
- `cfa.observability` → **`cfa.obs`**.

The full migration table and find-and-replace snippet are in the
[CHANGELOG](https://github.com/marquesantero/cfa/blob/main/CHANGELOG.md#11-0).

### Per-framework adapter shims removed

`cfa.adapters.langgraph`, `crewai`, `autogen`, `dspy`, and `openai_agents`
were 15-line files that all re-exported the same `cfa_guard` decorator.
They are removed. Use `cfa.adapters.cfa_guard` directly with any
framework. The full reasoning is in
[ADR-0006](https://github.com/marquesantero/cfa/blob/main/docs/adr/0006-no-fake-adapters.md).

### Hard-coded fiscal/Portuguese vocabulary extracted

The "rule-based production baseline" normalizer used to carry hard-coded
Brazilian fiscal terms in its defaults. That's now an explicit opt-in:

```python
from cfa.resolve import RuleBasedNormalizerBackend

backend = RuleBasedNormalizerBackend(
    domain_keywords={
        "fiscal_data_processing": ["nfe", "nota fiscal", "fiscal", "tribut"],
    },
)
```

A complete example with NF-e, SPED, CPF/CNPJ, "ouro"/"prata" layer
synonyms, and a `build_fiscal_pt_br_backend()` factory ships at
[`examples/fiscal_pt_br_normalizer.py`](https://github.com/marquesantero/cfa/blob/main/examples/fiscal_pt_br_normalizer.py).

### Perf baselines

`tests/perf/` is a new opt-in suite. Skipped by default; enable with
`pytest --run-perf` or `CFA_RUN_PERF=1`. The two tests cover:

- **`cfa_guard` call overhead** — p99 of a guarded call on a warm
  kernel. Measured ~2.4 ms on the reference machine; asserts ≤ 10 ms.
- **`cfa.testing.evaluate` throughput** — full evaluations per second
  with the rule-based normalizer. Measured ~930 ops/sec; asserts ≥ 20.

If a future change regresses either by more than ~3×, CI will catch it.

### Site rewrite

- The homepage hero now reads "Decide before you write." instead of
  "Technical Whitepaper v1.0.0."
- The whitepaper dropped its Greek-letter `(Φ, Γ, Π, Ω, Σ)` framing for
  plain operational English. The formal version is preserved at
  `drafts/WHITEPAPER_FORMAL.md` for whoever wants it.
- A new [Compare](/docs/compare) page explains where CFA pairs with
  (rather than replaces) OPA, LangSmith / Phoenix / Patronus, Great
  Expectations / Soda, Unity Catalog / Atlan / DataHub, and dbt
  contracts.
- Status banner on every page.

## On semver

1.1.0 carries breaking changes. That's normally a 2.0 thing.

Two reasons we chose 1.1.0:

1. 1.0.0 had no actual external adopters. The renames cost no real
   user anything.
2. The five distinctive primitives — the contracts that genuinely matter
   — are unchanged. The renames are editorial, not architectural.

Strict semver (breaking changes only in majors) resumes the moment
adoption appears. The
[roadmap](https://github.com/marquesantero/cfa/blob/main/drafts/ROADMAP.md)
makes that explicit.

## What's next (1.2.0)

The first real integration: `cfa dbt check`. It will:

1. Read `target/manifest.json` from a dbt project.
2. Derive a `StateSignature` per materialized model.
3. Run the active policy bundle against each.
4. Exit non-zero in CI when any model triggers a `BLOCK`.

Tracking issue + design notes will land in the GitHub project board
shortly.

## Migrating from 1.0.0

If you imported any of these:

```python
from cfa.normalizer.base import IntentNormalizer
from cfa.resolution import IntentNormalizer
from cfa.governance import PolicyEngine, StaticValidator
from cfa.validation.static import StaticValidator
from cfa.observability.promotion import PromotionEngine
from cfa.adapters.langgraph import cfa_guard
```

Replace with:

```python
from cfa.resolve import IntentNormalizer
from cfa.policy import PolicyEngine, StaticValidator
from cfa.validate.static import StaticValidator
from cfa.obs.promotion import PromotionEngine
from cfa.adapters import cfa_guard
```

Full table in the
[CHANGELOG](https://github.com/marquesantero/cfa/blob/main/CHANGELOG.md).
