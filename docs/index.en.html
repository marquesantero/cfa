CFA v2

An architecture for AI-native systems that replaces the static agents-and-skills model with intent resolution, governed planning, validated execution, explicit state projection, and evidence-based lifecycle evolution.

Entry Point

1. Overview

CFA starts from a simple thesis: an AI system should not jump directly from prompt to action. It should first understand the intent, verify whether that intent can be executed under the governing rules, plan under control, execute with validation, and explicitly update the state of the environment.

The result is a governed execution kernel: less coupled to static catalogs, more auditable, more predictable, and better suited for production in critical domains.

Section 02

2. Proposal

Instead of asking “which agent or skill should handle this?”, CFA asks “what state change is being requested, in which context, under which constraints, and can it be executed safely?”.

Semantic Resolution
Intent stops being loose text and becomes a typed StateSignature with domain, target, datasets, constraints, and execution context.
Governance Before Execution
The Policy Engine decides whether the intent is approved, replanned, or blocked before any execution touches real data.
State as Contract
The Context Registry closes the loop between what was executed and what may be executed next, avoiding implicit memory and environmental inconsistency.
Section 03

3. Project

This repository combines two things at once: the architectural proposal behind CFA and a concrete Python implementation of the core kernel. The goal is not only to describe an elegant idea, but to test it as a modular system with explicit contracts, audit persistence, state projection, and lifecycle evolution.

Current Scope
The project already covers intent normalization, policy, planner, code generation, static validation, sandbox execution, runtime validation, partial execution, state projection, audit trail, and lifecycle.
Section 04

4. Applications

Data Platforms
Data pipelines that require governance, controlled mutability, traceability, and publication consistency.
AI Ops
Flows that require deterministic execution, generated-code validation, and strong observability before production.
Regulated Domains
Scenarios with PII, schema contracts, audit requirements, and the need to explain why the system blocked or approved something.
Section 05

5. Implementation

The current CFA implementation is organized as a modular library in src/cfa, with a main orchestrator and independent submodules for different stages of the architectural contract.

Core Modules
resolution: intent interpretation and confirmation.
governance: rules and pre-execution validation.
lifecycle: indices, promotion, and demotion.
kernel: full end-to-end orchestration.
Implemented Components
normalizer.py, policy.py, planner.py, codegen.py, static_validation.py, sandbox.py, runtime_validation.py, partial_execution.py, state_projection.py, audit.py, promotion.py.

Implemented Flow

Natural language → resolution → confirmation → policy → planning → code generation → static validation → execution → runtime validation → partial execution → state projection → audit → lifecycle
